# 間取りアーキテクト Pro (roomGenerator) - システム分析とリファクタリング計画

本ドキュメントは、コントリビュータ向けに `roomGenerator` のシステム構造、主要機能、データフロー、およびワークフローを詳細に解説するものです。今後のリファクタリングの指針としても機能します。

## 1. プロジェクト概要
`roomGenerator` は、Wails (Go) と React (Vite) を用いて構築された、間取り図（部屋のレイアウト）およびカスタムアセット（家具・設備など）を生成・編集するためのデスクトップアプリケーションです。

## 2. アーキテクチャ全体像
システムは主に以下の3つの層で構成されています。

- **フロントエンド (React + Vite):**
  - ユーザーインターフェースの描画、キャンバス操作（ドラッグ、リサイズ等）、状態管理を担当します。
  - ルーティングには `react-router-dom` (HashRouter) を使用しています。
- **状態管理 (Zustand):**
  - アプリケーションのグローバルな状態（プロジェクトデータ、キャンバスの状態、UI状態）を一元管理します。
  - `zundo` ミドルウェアにより、操作履歴（Undo/Redo）機能を提供します。
- **バックエンド (Go + Wails):**
  - OSネイティブな機能へのアクセス、ファイルシステムへのプロジェクトデータの読み書き（JSON形式）を担当します。

## 3. URL・ルーティング構造
アプリケーションは以下のURLパターンでルーティングされます。

| URLパターン | コンポーネント (`frontend/src/pages/`) | 役割 |
|---|---|---|
| `/` | `Home.jsx` | プロジェクト一覧を表示するランディングページ。 |
| `/library` | `Library.jsx` | グローバルアセット（共有パーツ）の管理画面。 |
| `/project/:id` | `Editor.jsx` | 特定のプロジェクトを編集するメインのエディタ画面。 |
| `/settings` | `Settings.jsx` | アプリケーション全体の設定（カラーパレット等）画面。 |

## 4. 主要コンポーネント構成と役割
エディタ画面 (`/project/:id`) を構成する主要なコンポーネントは以下の通りです。

- **`Editor.jsx`:** メインコンポーネント。現在のモード（Layout / Design）に応じて表示を切り替えます。
  - **`UnifiedSidebar.jsx`:** 左側パネル。アセットの選択、ツールの切り替えを行います。
  - **`LayoutCanvas.jsx`:** 「レイアウトモード」用のキャンバス。アセットをインスタンスとして配置し、間取りを作成します。
  - **`DesignCanvas.jsx`:** 「パーツ設計モード」用のキャンバス。個別の図形（ポリゴン、円、矩形）を組み合わせてアセットを作成・編集します。
  - **`LayoutProperties.jsx` / `DesignProperties.jsx`:** 右側パネル。選択されたインスタンスまたは図形のプロパティ（サイズ、色、座標など）を編集します。
  - **キャンバスのロジック分離:** ドラッグ操作やリサイズなどの複雑なインタラクションは、各キャンバスコンポーネント内のロジックファイル（例：`DesignCanvas.logic.js`）やカスタムフック（例：`useCanvasInteraction.js`）に分離されています。

## 5. データフローと状態管理 (Zustand)
状態管理は機能ごとにスライス（分割）され、`frontend/src/store/index.js` で結合されています。

- **主要なスライス (`frontend/src/store/`):**
  - `projectSlice.js`: プロジェクトの基本情報、カメラ状態（`viewState`）を管理。
  - `assetSlice.js`: プロジェクト固有のアセット（`localAssets`）やグローバルアセットを管理。
  - `instanceSlice.js`: レイアウト上に配置されたインスタンス（`instances`）を管理。
- **データ永続化の仕組み:**
  - ユーザーがキャンバス上で操作を行うと、Zustandのストアが即座に更新されます。
  - `useAutoSave` フックがストアの変更を検知し、自動的にバックエンド (`API.saveProjectData`) を呼び出してファイル（`data/project_{id}.json`）に保存します。

## 6. 主要な関数とドメインロジック
ビジネスロジックや幾何学計算は、UIコンポーネントから分離されています。

- **`frontend/src/domain/geometry.js`:**
  - 座標変換、回転（`rotatePoint`）、SVGパスの生成（`generateSvgPath`）などの純粋な数学的計算。
  - Bounding Box（AABB）の計算（`getRotatedAABB`）を行い、アセット全体のサイズと中心座標を決定します。
- **`frontend/src/domain/assetService.js`:**
  - `updateAssetEntities(asset, entities)`: アセット内の図形（Entities）を更新し、全体のAABBサイズを再計算して一貫性を保ちます。アセット操作の標準的な手法です。
  - `forkAsset(asset, defaultColors)`: グローバルアセットをプロジェクト用に複製（ディープコピー）し、新しいIDを付与します。
- **`frontend/src/domain/projectService.js`:**
  - プロジェクトデータの読み込み時に、古いデータ形式（例：`shapes` プロパティ）を新しい形式（`entities`）に正規化（マイグレーション）します。

## 7. ワークフロー（プロジェクトの保存から参照まで）
1. **読み込み:**
   - ユーザーが `/project/:id` にアクセス。
   - `Editor.jsx` がマウントされ、Zustandの `loadProject(id)` を呼び出します。
   - Goバックエンド（`GetProjectData`）からJSONファイルが読み込まれ、フロントエンドで正規化後、Zustandストアに格納されます。
2. **編集と状態の同期:**
   - 高頻度な更新（ドラッグ中のプレビュー等）は Reactのローカルステート (`localAsset` など) で管理し、パフォーマンスを維持します。
   - 操作完了時（`onPointerUp`等）にのみ Zustand ストアを更新し、履歴（Undo/Redo）に追加します。
3. **保存:**
   - Zustandストアが更新されると `useAutoSave` が検知し、Goの `SaveProjectData` を経由してローカルのJSONファイルに永続化されます。

## 8. 今後のリファクタリング方針
現在のアーキテクチャは十分に機能していますが、今後の保守性向上のため、以下の領域でリファクタリングを検討します。

1. **コンポーネントの責務分割:**
   - `Editor.jsx` や `DesignCanvas.jsx` など、まだ肥大化しているコンポーネントから、純粋な描画機能とビジネスロジックをさらに分離する（例：カスタムフックへの抽出）。
2. **型の安全性向上:**
   - 現状はJSDocによる `@typedef` で型を定義していますが、重要なデータモデル（`Asset`, `Entity`, `Instance`）については、フロントエンドでの型チェックをより厳格に行う仕組み（あるいはTypeScriptへの移行）を検討する。
3. **状態管理の最適化:**
   - Zustandのスライス間で依存関係が複雑化している部分を整理し、単方向データフローをより明確にする。
4. **テストカバレッジの向上:**
   - Playwrightを用いたE2Eテスト、およびGoバックエンドのユニットテスト（特にデータ移行ロジック）を拡充し、リファクタリング時の安全性を担保する。
