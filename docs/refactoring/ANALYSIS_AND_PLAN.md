# roomGenerator システム分析・リファクタリング計画書

このドキュメントは、新規コントリビューターが `roomGenerator` の全体構造を把握し、どこで何が処理され、データがどのように保存・参照されているかを理解するためのガイドです。
また、将来的なリファクタリングの指針としても機能します。

## 1. 全体アーキテクチャ概要

`roomGenerator` は、Wailsフレームワークを使用したデスクトップアプリケーションです。
- **フロントエンド**: React (Viteビルド)
- **バックエンド**: Go
- **通信**: Wailsによる関数バインディング (JSONベースのシリアライズ)
- **状態管理**: Zustand (+ zundo による Undo/Redo)
- **ルーティング**: React Router (`HashRouter`)

## 2. ディレクトリ構造と主要コンポーネント

プロジェクトの主要なディレクトリとファイルは以下の通りです。

```text
roomGenerator/
├── app.go              # バックエンドのメインロジック (ファイルI/O, APIエンドポイント)
├── models.go           # Goのデータ構造体定義 (型安全性の担保)
├── frontend/           # フロントエンド (React)
│   ├── src/
│   │   ├── pages/      # 画面単位のコンポーネント (ルーティングに対応)
│   │   ├── components/ # 再利用可能なUI・キャンバスコンポーネント
│   │   ├── store/      # Zustandによるグローバル状態管理
│   │   ├── domain/     # ドメインロジック (アセット操作, プロジェクト読み込み等)
│   │   ├── lib/        # 汎用ユーティリティ (API通信, 座標変換, 共通定数)
│   │   └── App.jsx     # ルートコンポーネント
└── data/               # ローカルに保存されるJSONデータ (Git管理外)
```

## 3. URL（ルーティング）と画面 (frontend/src/pages)

アプリケーションは `HashRouter` を使用して画面遷移を行います。

| URLパス | コンポーネント | 役割 |
|---|---|---|
| `/` | `Home.jsx` | ホーム画面。既存プロジェクトの一覧表示、新規作成を行います。 |
| `/library` | `Library.jsx` | グローバルライブラリ画面。共通で使用するアセットやカラーパレットの管理を行います。 |
| `/project/:id` | `Editor.jsx` | メインの編集エディタ画面。プロジェクトの読み込みと各種キャンバスのオーケストレーションを行います。 |
| `/settings` | `Settings.jsx` | アプリケーション全体の設定画面（将来拡張用）。 |

## 4. 主要なコンポーネントの役割 (frontend/src/components)

`Editor.jsx` の配下には、以下の主要なコンポーネントが配置されています。

- **`UnifiedSidebar.jsx`**: 画面左側のサイドバー。アセットの一覧表示や、配置・編集ツールの切り替えを行います。
- **`LayoutCanvas.jsx`**: 「レイアウトモード」用のキャンバス。アセットをインスタンスとして配置し、間取り図を作成します。
- **`DesignCanvas.jsx`**: 「デザインモード」用のキャンバス。個別のアセット（家具や部屋の形状）を頂点ベースで編集・作成します。複雑な操作ロジックは `DesignCanvas.logic.js` に分離されています。
- **`LayoutProperties.jsx` / `DesignProperties.jsx`**: 画面右側のプロパティパネル。選択されたオブジェクト（インスタンスまたは形状）のプロパティ（色、サイズ、角度など）を編集します。

## 5. データモデルと状態管理 (frontend/src/store)

状態管理には `Zustand` が使われており、機能ごとにスライス分割されています。

### 5.1. 主要な状態 (State)
- **`projects` (`projectSlice.js`)**: 全プロジェクトのメタデータ一覧。
- **`localAssets` (`assetSlice.js`等)**: 現在開いているプロジェクト固有のアセット定義（形状データを含む）。
- **`instances` (`instanceSlice.js`)**: LayoutCanvas 上に配置されたアセットの実体（座標、角度情報を持つ）。
- **`viewState` (`projectSlice.js`)**: キャンバスのパン（移動）とズームの表示状態。

### 5.2. データの参照と更新の流れ（Workflow）

Zustandのストアを通じてコンポーネント間で状態が共有されます。コンポーネントは必要な状態のみを `useStore` フックで購読(`subscribe`)し、状態が変更された際に再レンダリングされます。

**【データの流れの例：オブジェクトの移動】**
1. ユーザーがキャンバス上のオブジェクトをドラッグする。
2. `DesignCanvas` または `LayoutCanvas` がマウスイベントを検知し、一時的なローカル状態（React State）を更新して高速に再描画する。
3. ドラッグ終了時（`onPointerUp`）に、確定した座標を Zustand ストアの更新アクション（例: `updateAssetEntities` または `updateInstance`）に渡す。
4. Zustand ストアが更新される。
5. `zundo` ミドルウェアが変更履歴を保存する（Undo用）。
6. カスタムフック `useAutoSave` がストアの変更を検知し、Goバックエンドに変更データを送信する。

## 6. データの保存と永続化 (Backend / File System)

### 6.1. 保存のタイミング
フロントエンド側で `hooks/useAutoSave.js` が Zustand ストア内の保存対象データ（`localAssets`, `instances` 等）を監視しています。変更が一定時間（デバウンス処理）検知されない場合に、自動的に `API.saveProjectData` が呼び出されます。

### 6.2. バックエンド処理 (app.go & models.go)
1. フロントエンドから Wails を経由して `(a *App) SaveProjectData(id, data)` が呼び出されます。
2. 送信された `data` は一度 JSON にマーシャル（シリアライズ）されます。
3. 次に、`models.go` で定義された厳密な型を持つ `ProjectData` 構造体にアンマーシャルされます。このステップで型の検証や不正なフィールドのフィルタリングが行われます（型安全性の担保）。
4. 検証されたデータが、`data/project_{id}.json` ファイルに保存されます。

## 7. リファクタリングの方針・課題点

現状の構成における今後のリファクタリング方針は以下の通りです。

1. **ドメインロジックのさらなる分離**:
   現在、一部のコンポーネント内に複雑な計算ロジックが残っている可能性があります。座標計算や図形のバウンディングボックス計算（AABB）などは、`frontend/src/domain/geometry.js` や `utils.js` に集約し、コンポーネントは純粋なView層としての役割に徹するようにします。
2. **パフォーマンスの最適化**:
   Zustand の不要な再レンダリングを防ぐため、`useStore` で状態を取得する際のセレクタをより細かく指定するか、`shallow` 比較を適切に導入します。
3. **テストカバレッジの向上**:
   Goバックエンドの `app_test.go` は初期的なものです。データマイグレーションロジック（`normalizeProjectData`）や、ファイルの読み書き処理に対するユニットテストを拡充します。
4. **型定義の導入 (フロントエンド)**:
   将来的には、React側のコードに TypeScript または JSDoc による型注釈を導入することで、Goバックエンドの `models.go` との型不一致によるバグを防ぎます。
